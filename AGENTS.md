# ACTI リポジトリ Codex 動作ルール

このファイルは、このリポジトリを別のPCで開いた Codex にも適用する絶対ルールである。
ACTI は有料サービスであり、顧客への案内より検証の完了を優先する。

## 1. 完了の定義

- 実装、ローカル検証、GitHub CI、本番デプロイ、本番検証、監視、テストデータ削除の全工程が終わるまで作業を続ける。
- 「コードを書いた」「ビルドが通った」「デプロイした」は完了ではない。
- ユーザーから追加の催促がなくても、未完了の工程があれば Codex が自律的に続行する。
- ユーザーの指摘をテスト工程の代わりにしない。変更箇所、隣接経路、異常系、実ブラウザ、本番保存、自動監視まで先回りして確認する。
- 将来の障害ゼロは保証しない。ただし、確認可能な失敗経路を残したまま「修正済み」「復旧済み」「問題ない」と報告しない。
- Codex が確認できる操作をスタッフへ差し戻さない。人間にしかできない認証操作だけを、一度で具体的に依頼する。

## 2. 作業開始前

1. `git status`、直近コミット、本番コミット、未コミット差分を確認し、他者の変更を上書きしない。
2. 関連するサポートチケット、監視結果、Vercelログ、Supabase状態、過去の修正履歴を確認する。
3. 症状、再現条件、影響範囲、根本原因、同じ原因を共有する隣接経路を整理する。
4. 正常系と異常系のテスト表を作る。最低限、入力上限、長い履歴、連続操作、同時実行、通信切断、タイムアウト、外部API失敗、保存失敗、再読み込みを含める。
5. 不具合を自動テストで再現できる場合は、修正前に失敗する回帰テストを作る。

## 3. 実装ルール

- 既存の設計、型、バリデーション、エラー形式、テスト方式を優先する。
- エラーを隠すfallbackだけで終わらせず、根本原因、再試行、保存整合性、利用者への復旧導線まで扱う。
- 同じ障害を起こした経路と、その経路を呼ぶ別画面・別APIも確認する。
- 認証、権限、RLS、個人情報、決済、メール送信を変更する場合は、権限の不足と過剰の両方をテストする。
- 一括更新で `subscription_status='cancelled'` を有効へ戻さない。
- AWAKES / MyASP のCSVは単一エンコーディングで決め打ちせず、UTF-8、SHIFT_JIS、CP932等の件数を比較する。CP932を必ず含める。
- 秘密鍵、APIキー、webhook secret、顧客の個人情報をコード、ログ、PR、公開ファイルへ入れない。

## 4. ローカルゲート

対象変更に該当する項目をすべて実行する。省略する場合は「対象外の理由」を記録する。

1. `npm ci`
2. `npm run verify:coaching:ci`
3. 認証、権限、入力検証、境界値、重複送信、冪等性、並行実行を確認する。
4. timeout、再試行、途中で終わるstream、DB保存失敗、通信切断後の復旧を確認する。
5. PCとスマートフォンの実ブラウザで主要操作を行う。
6. AIコーチング変更時は、日本語IME変換中Enter、通常Enter、Shift+Enter、連続送信、二重クリック、長文、長い履歴、添付上限、再読み込み、JavaScript例外、表示崩れを必ず確認する。
7. テストが一つでも失敗したら、原因を修正して回帰テストを追加し、全テストを最初から再実行する。

## 5. GitHubゲート

- `main` へ直接pushしない。作業ブランチからPRを作る。
- 必須CI `unit-and-build` が成功するまでマージしない。
- CIのスキップ、失敗の無視、ブランチ保護の解除は禁止する。
- 緊急障害でも検証を省略しない。必要ならテストを並列化する。
- PRの差分に意図しないファイル、秘密情報、テスト生成物がないことを確認する。

## 6. 本番ゲート

1. マージ済みGit SHAとVercel本番デプロイのSHAが一致し、Production AliasがReadyであることを確認する。
2. 本番URLに対して `npm run verify:coaching:production` を実行する。
3. stream完了、初回応答時間、総応答時間、保存、再読み込み、連続会話、長い履歴、同時接続、添付を確認する。
4. Vercel、Supabase、クライアント障害ログを確認し、テスト時間帯に未説明の5xx、timeout、stream未完了、DBエラーがないことを確認する。
5. テスト用Authユーザー、profiles、chat_sessions、chat_messages、Storage添付が削除されたことを件数で確認する。
6. 定期監視は手動実行だけで済ませない。実際のスケジュール起動、DB保存、失敗通知先まで確認する。

## 7. 顧客案内前の観測ゲート

顧客影響のある不具合について「修正済み」「復旧済み」「もう使える」と案内できるのは、次をすべて満たした後だけである。

1. ローカル、GitHub、本番の全ゲートが成功している。
2. 10分間隔の自動本番監視が20分以上にわたり3回連続で成功している。
3. 同じ観測期間に、未説明の5xx、クライアント障害通知、同症状の新規サポートチケットがない。
4. 変更箇所以外から同じ症状が発生しないか再監査し、中または高リスクの未検証項目がゼロである。

観測中の報告は「追加対策を本番へ反映し、検証と監視を継続しています」とする。「修正済み」「復旧しました」と言わない。

## 8. 完了時の証拠

完了報告には以下を必ず含める。

- PR番号、マージSHA、VercelデプロイID
- 実行コマンド、テスト総数、成功数、実行日時
- 自動監視3回分の日時、所要時間、DB保存行
- ログを確認した時間帯、5xx・timeout・stream未完了・DBエラーの件数
- テストデータ削除後のAuth、DB、Storage件数
- 未確認事項。ゼロなら「未確認事項なし」と明記する

証拠が揃う前に完了報告を出さない。途中経過では、完了した工程と未完了の工程を区別して伝える。
